# Ethical Use Cases

Open Defense Kit is designed for authorized security education, defensive review, and controlled lab work.

## Allowed use cases

### 1. Personal security learning

A learner can use the project to understand how security tools are categorized, what they are used for, and what risks they carry.

Allowed examples:

- learning what recon means
- learning the difference between OSINT and exploitation
- studying defensive scanning workflows
- using local intentionally vulnerable labs

### 2. Owned asset review

A developer or founder can review their own domains, repositories, servers, and applications.

Allowed examples:

- checking security headers on your own website
- scanning your own code for secrets
- reviewing dependencies for known vulnerabilities
- checking container images before deployment
- checking cloud misconfiguration in your own account

### 3. Blue-team practice

Security learners can use Open Defense Kit to understand attacker patterns from a defender's perspective.

Allowed examples:

- building detection notes
- mapping risky tool behavior to defensive controls
- documenting logs generated by security tools
- creating incident response checklists

### 4. Classroom or workshop labs

Educators can use the toolkit to demonstrate safe security concepts in isolated environments.

Allowed examples:

- local VM labs
- capture-the-flag environments
- intentionally vulnerable web apps
- offline packet analysis
- malware-free reverse engineering exercises

### 5. Internal company audit support

A company can use the project as a checklist and helper toolkit for internal security hygiene.

Allowed examples:

- dependency audit workflow
- repository hygiene checks
- exposed-secret detection
- cloud posture review
- asset inventory

## Disallowed use cases

Open Defense Kit must not be used for:

- accessing systems without permission
- phishing real people
- stealing credentials
- attacking live third-party networks
- disrupting services
- building malware or payloads
- operating RATs, botnets, or C2 infrastructure
- bypassing authentication on systems you do not own
- spying through cameras, microphones, or private accounts
- harassment, intimidation, extortion, or illegal surveillance

## Permission standard

Before testing anything, the user should be able to answer:

1. Do I own this target?
2. If not, do I have written permission?
3. Is the scope clear?
4. Is the test environment safe?
5. Could this disrupt someone else's service?
6. Are logs and results stored responsibly?

If any answer is unclear, do not run the test.

## Safe target examples

- `localhost`
- your own lab VM
- intentionally vulnerable apps such as DVWA/Juice Shop in a local lab
- your own domain
- your own repository
- your own cloud account
- a CTF platform that explicitly allows testing

## Unsafe target examples

- random websites
- public IP ranges you do not own
- school/work networks without written permission
- social media accounts
- banking systems
- government systems
- telecom infrastructure
- other people's cameras, routers, phones, or accounts
