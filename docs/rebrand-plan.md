# Rebrand Plan

This project is being repositioned from a risky hacking-tool launcher into Open Defense Kit: a legal, ethical, defense-first security learning and audit toolkit.

## New identity

**Name:** Open Defense Kit  
**Command:** `odkit`  
**Tagline:** Defensive security learning and auditing, safely organized.  
**Audience:** Developers, founders, students, blue-team learners, internal security teams.

## Old identity to remove

Remove language such as:

- hacking tools
- hack camera
- password attack
- exploit people
- social engineer victims
- attack Wi-Fi
- DDoS
- payload builder
- RAT
- crack accounts

## New language to use

Use language such as:

- authorized security learning
- defensive audit
- lab-only testing
- owned assets
- security hygiene
- blue-team practice
- vulnerability awareness
- safe recon
- local lab
- permission-based checks

## Category mapping

| Legacy category | New treatment |
|---|---|
| Web hacking | Web security checks for owned apps |
| Password attack | Password policy education / local lab only |
| Exploitation tools | Vulnerability learning labs only; no live-target exploitation |
| Wireless testing | Isolated wireless lab notes only |
| Social engineering | Awareness training only; no phishing automation |
| Camera hacking | Remove |
| Web information | Defensive OSINT for owned domains |
| Update/uninstall | Replace with safe package workflow |

## Visual direction

Avoid skulls, weapons, anonymous masks, red-alert hacker visuals, or aggressive terminal art.

Preferred feel:

- clean terminal UI
- professional security dashboard tone
- blue/green defensive palette
- clear module cards
- risk labels
- permission warnings
- audit log summaries

## README positioning

The README should answer:

1. What is this?
2. Who is it for?
3. What is allowed?
4. What is prohibited?
5. Why is it safer than old hacking tool collections?
6. How do I use it in a lab?
7. How do I contribute safely?

## Initial public promise

> Open Defense Kit organizes security learning and defensive checks into a safer, permission-based workflow. It does not exist to help users attack systems. It exists to help users understand, audit, and defend systems they are allowed to test.
